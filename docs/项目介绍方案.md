# 工业设备故障预测与健康评估系统项目介绍方案

## 1. 项目名称

**基于工业物联网大数据的设备故障预测与健康评估系统**

英文名称可表述为：

**Design and Implementation of Equipment Fault Prediction and Health Evaluation System Based on Industrial Internet of Things Big Data**

## 2. 项目定位

本项目面向工业设备预测性维护场景，围绕设备传感器数据构建一套从数据接入、数据治理、特征窗口、模型推理、风险规则、预警事件到前端可视化展示的闭环系统。

系统不是单纯展示数据的看板，而是一个强调“可信解释链路”的工业预测系统。每一次预警都能够追溯到对应的预测记录、特征窗口、模型版本、故障概率、健康评分和主要贡献特征，避免只用“模型算出来的”这种不可解释说法。

## 3. 项目背景

在工业生产中，机床、产线设备、旋转机械等关键设备长期处于连续运行状态。一旦发生故障，可能造成停机、产能损失、维修成本上升，甚至引发安全风险。

传统维护方式主要有两类：

1. **事后维修**：设备故障后再维修，停机损失高。
2. **定期保养**：按周期维护，容易过度维护或漏掉突发风险。

预测性维护的目标是利用设备运行过程中的传感器数据，提前识别设备健康状态变化，预测故障风险，并给出可解释的预警和维护建议。

本项目基于工业物联网数据，构建故障预测、异常检测、健康评分和预警管理系统，用于辅助运维人员提前发现高风险设备，提升设备运维决策效率。

## 4. 项目目标

项目目标可以概括为四句话：

1. **接得进来**：支持历史 CSV 数据导入，也支持实时设备读数接入。
2. **算得清楚**：对原始读数进行治理、窗口化和特征构造，再进入模型推理。
3. **解释得明白**：输出故障概率、健康评分、风险等级、RUL、SHAP 贡献因子和规则解释。
4. **闭环管得住**：预警不是静态消息，而是具备确认、处理、关闭、忽略等生命周期状态。

## 5. 核心业务流程

系统主链路如下：

```text
历史数据 / 实时读数
  → 数据接入
  → 数据治理
  → 特征窗口
  → 模型版本
  → 故障预测
  → 风险规则
  → 预警事件
  → 生命周期处理
  → 前端可视化展示
```

更具体的数据闭环如下：

```text
CSV 导入 / HTTP 实时接入 / WebSocket 实时接入
  → sensor_readings 原始读数
  → feature_windows 特征窗口
  → model_versions 模型版本
  → prediction_logs 预测记录
  → prediction_explanations 解释因子
  → warning_events 预警事件
  → warning_action_logs 处理记录
```

## 6. 系统整体架构

当前系统采用前后端分离架构。

```text
前端 React 控制台
  ↓ REST API / WebSocket
FastAPI 后端服务
  ↓
业务服务层 / 模型推理层 / 数据治理层
  ↓
MySQL 数据库
  ↓
设备、点位、读数、特征、预测、解释、预警、模型版本
```

当前项目也预留了 Kafka、MQTT、Redis 等准生产链路能力，但现阶段核心闭环以 FastAPI + MySQL + React 为主。

## 7. 模块划分

| 模块 | 作用 |
|---|---|
| 数据接入模块 | 支持 AI4I CSV 导入、HTTP 实时读数接入、WebSocket 实时读数接入 |
| 设备主数据模块 | 管理设备台账、工厂、车间、产线、设备状态 |
| 点位主数据模块 | 管理传感器点位、单位、采样频率、量程、启停状态 |
| 数据治理模块 | 对读数进行标准化、质量控制、窗口化前处理 |
| 特征工程模块 | 生成均值、标准差、最大值、最小值、趋势等窗口特征 |
| 模型分析模块 | 完成故障分类、异常检测、RUL 预测、健康评分计算 |
| 风险规则模块 | 将故障概率和健康评分转换为 low / medium / high / critical 风险等级 |
| 预警中心模块 | 生成预警事件，支持确认、处理、关闭、忽略和动作留痕 |
| 模型管理模块 | 展示模型版本、模型类型、指标、状态和产物路径 |
| 可视化模块 | 展示总览、实时监测、设备管理、预测记录、预警中心和模型管理 |

## 8. 后端技术方案

后端使用 FastAPI 构建 API 服务，核心职责是接收数据、执行业务流程、调用模型推理、写入数据库并向前端提供查询接口。

### 8.1 后端分层

```text
backend/app
  ├─ api              # API 路由层
  ├─ compute          # 特征计算层
  ├─ governance       # 数据治理层
  ├─ ingestion        # 数据导入和数据转换
  ├─ models           # 模型训练、预测、风险规则、健康评分
  ├─ repositories     # 数据访问层
  ├─ schemas          # 数据结构定义
  └─ services         # 业务服务和预测流水线
```

### 8.2 后端核心接口

| 接口方向 | 说明 |
|---|---|
| 健康检查 | 检查后端服务可用性 |
| CSV 导入 | 上传 AI4I 数据集，训练模型并可选择回放生成预测记录 |
| 实时读数接入 | 真实设备通过 MQTT→EMQX→Kafka 主链路接入，HTTP/WS 仅作为调试或旁路适配入口 |
| 设备查询 | 查询设备台账、最新健康评分、风险等级和点位列表 |
| 预测查询 | 查询预测记录、故障概率、健康评分、解释因子 |
| 预警查询 | 查询预警事件、风险等级、处理状态、追溯链路 |
| 预警状态更新 | 执行确认、处理、关闭、忽略等状态迁移 |
| 模型查询 | 查询模型版本、模型指标和 active 模型状态 |

## 9. 前端技术方案

前端使用 React + TypeScript 实现工业控制台风格界面，强调信息密度、状态解释和闭环可视化。

### 9.1 页面组成

| 页面 | 说明 |
|---|---|
| 系统总览 | 展示设备总数、异常设备、高风险设备、平均健康评分、今日预警 |
| 实时监测 | 面向真实工业设备 MQTT 接入链路展示实时预测、在线状态和风险变化；展示适配器仅用于联调 |
| 数据导入 | 支持 AI4I CSV 上传、模型训练、训练展示数据回放 |
| 设备管理 | 展示设备台账、最新健康评分、风险等级、在线/离线/检修/故障状态、点位列表 |
| 预测记录 | 展示故障概率、异常分数、健康评分、RUL、SHAP Top 贡献因子 |
| 故障与预警中心 | 展示预警事件、规则解释、追溯链路、维护建议和生命周期操作 |
| AI 模型管理 | 展示模型族、模型指标、模型版本、产物路径和状态 |

### 9.2 前端设计原则

1. **不伪造数据**：接口为空时明确展示空态，不伪装成真实监控数据。
2. **状态优先**：风险等级、设备状态、预警状态使用文字与颜色同时表达。
3. **闭环展示**：预警页不仅展示结果，还展示预测 ID、窗口 ID、模型版本和解释因子。
4. **高密度但不堆叠**：列表、预警、模型版本均支持分页，避免大量数据堆叠。
5. **展示接入不中断**：启动展示适配器后切换页面不会停止，除非用户手动暂停；真实设备接入由 MQTT→EMQX→Kafka 后台链路持续承接。

## 11. 可信预警闭环设计

本项目最重要的可信闭环是：每条预警都不是孤立文本，而是绑定到一次具体预测。

```text
warning_event
  ├─ prediction_id          # 对应预测记录
  ├─ device_code            # 设备编码
  ├─ feature_window_id      # 对应特征窗口
  ├─ model_version          # 使用的模型版本
  ├─ failure_probability    # 故障概率快照
  ├─ health_score           # 健康评分快照
  ├─ risk_level             # 风险等级
  └─ warning_explanation    # 解释因子快照
```

这样答辩或实际使用时，如果有人问“为什么这台设备是严重风险”，系统可以回答：

```text
因为该预警绑定的预测记录显示：
- 故障概率达到高风险阈值；
- 健康评分低于安全区间；
- 主要贡献特征包括 Tool wear、Torque 等；
- 该判断来自指定模型版本和指定特征窗口。
```

## 12. 模型方案

当前模型围绕 AI4I 2020 Predictive Maintenance 数据集构建。

### 12.1 模型能力

| 能力 | 实现方式 |
|---|---|
| 故障分类 | LightGBM 分类模型 |
| 异常检测 | Isolation Forest |
| 剩余寿命预测 | LightGBM RUL 回归模型 |
| 健康评分 | 故障概率、异常分数、趋势因子、质量分融合 |
| 可解释性 | SHAP 贡献因子 |
| 风险分级 | 显式风险规则 |

### 12.2 输出结果

每次推理输出：

```text
prediction_result
  ├─ device_code
  ├─ failure_probability
  ├─ anomaly_score
  ├─ health_score
  ├─ risk_level
  ├─ rul_hours
  ├─ trend_factor
  ├─ quality_score
  └─ explanations
```

## 13. 风险规则设计

风险等级不是前端随意判断，而是在后端规则模块中统一定义。

```text
critical: failure_probability >= 0.8 OR health_score < 30
high:     failure_probability >= 0.6 OR health_score < 50
medium:   failure_probability >= 0.3 OR health_score < 75
low:      otherwise
```

前端展示时会把规则解释为可读文本，例如：

```text
严重风险
原因：
- 故障概率 0.87，超过严重阈值 0.80
- 健康评分 26.4，低于严重阈值 30
- 主要贡献特征：Tool wear、Torque
```

## 14. 预警生命周期设计

预警不是生成后就结束，而是进入状态机管理。

```text
new 新预警
  → acknowledged 已确认
  → processing 处理中
  → resolved 已关闭
  → ignored 已忽略
```

每次状态变化都会写入动作日志：

```text
warning_action_log
  ├─ warning_id
  ├─ from_status
  ├─ to_status
  ├─ operator
  ├─ note
  └─ created_at
```

这使系统具备工业运维系统应有的处理闭环，而不是单纯的预警列表。

## 15. 项目亮点

### 15.1 从演示看板升级为可信闭环系统

系统不仅展示预测结果，还保存完整的数据链路：原始读数、特征窗口、模型版本、预测记录、解释因子和预警事件。

### 15.2 风险规则显式化

风险等级不是散落在前端或多个业务函数中，而是集中在后端规则模块，便于解释、修改和答辩。

### 15.3 预警具备生命周期

支持新预警、确认、处理中、关闭、忽略等状态流转，并保存操作记录。

### 15.4 模型结果具备解释性

通过 SHAP 展示 Top 贡献因子，让用户知道哪些特征影响了故障预测。

### 15.5 主数据补齐

设备和传感器点位不再只是随着数据被动生成，而是形成设备台账和点位台账，具备工业系统的基础对象模型。

### 15.6 实时接入状态稳定

设备接入启动后，切换页面不会中断实时模拟流，符合控制台类系统的使用习惯。

### 15.7 页面分页和大数据展示友好

设备、点位、预测记录、预警事件、模型卡片、模型版本均已分页，避免数据堆叠。

## 18. 可用于答辩的简短介绍

本项目是一个面向工业物联网场景的设备故障预测与健康评估系统。系统以设备传感器数据为基础，支持历史 CSV 数据导入和实时读数接入，通过数据治理、特征窗口构建、LightGBM 故障分类、Isolation Forest 异常检测、RUL 预测和健康评分模型，输出设备故障概率、健康评分、风险等级和主要影响特征。

系统重点解决预测结果可信度问题。每条预警都绑定具体预测记录、特征窗口、模型版本、故障概率、健康评分和解释因子，能够说明“为什么产生这个预警”。同时，预警中心支持新预警、确认、处理中、关闭和忽略等生命周期操作，并保留处理日志，形成从数据接入到预警处理的完整闭环。

前端采用工业控制台风格，包含系统总览、实时监测、数据导入、设备管理、预测记录、故障与预警中心、AI 模型管理等页面，帮助运维人员快速识别高风险设备、理解风险原因并跟踪处理状态。

## 19. 可用于论文摘要的表述

针对传统设备维护方式存在故障响应滞后、维护决策依赖人工经验、预测结果缺少解释依据等问题，设计并实现了一套基于工业物联网大数据的设备故障预测与健康评估系统。系统以设备传感器数据为输入，构建了数据接入、数据治理、特征窗口、模型推理、风险规则、预警生命周期和可视化展示的完整闭环。模型层采用 LightGBM 进行故障概率预测，采用 Isolation Forest 进行异常检测，并结合健康评分和 RUL 预测对设备状态进行综合评估。系统通过 SHAP 贡献因子和显式风险规则增强预测结果解释性，并将每条预警与预测记录、特征窗口和模型版本绑定，实现预警级追溯。实验和系统实现表明，该系统能够有效支撑工业设备健康状态监测、故障风险识别和预测性维护决策。

## 20. 后续增强方向

当前系统已经完成可信闭环，后续如果要向真实工厂生产链路增强，可以继续推进：

1. **Kafka 真正启用**：拆分 raw、cleaned、features、predictions、warnings 多 Topic。
2. **EMQX 接入**：替代简单 MQTT Broker，支持设备连接管理、认证和规则引擎。
3. **TSDB 时序库**：将高频传感器读数迁移到 TimescaleDB / InfluxDB / ClickHouse。
4. **Redis 实时状态层**：保存设备最新状态、在线状态、幂等键和告警抑制。
5. **用户权限和审计**：补充角色、权限、操作审计和数据隔离。
6. **工单系统联动**：将预警升级为维修工单，形成维护执行闭环。
7. **模型发布治理**：支持模型审批、灰度发布、回滚和多版本对比。

## 21. 一句话总结

本项目的核心价值是：**把工业设备传感器数据转化为可解释、可追溯、可处理的故障预警闭环，帮助运维人员提前发现风险并做出维护决策。**
