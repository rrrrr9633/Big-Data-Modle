# 后端接口与功能总结

## 1. 后端总体定位

当前后端是一个基于 `FastAPI` 的工业设备预测性维护系统后端，统一接口前缀为：

```text
/api/v1
```

它已经不只是普通 CRUD API，而是同时承担三类职责：

```text
接口层：
  设备 / 仪表盘 / 预测 / 预警 / 模型 / 数据导入 / 遥测接入 / 实时监测

数据管道层：
  MQTT 接入 / Kafka 缓冲 / 数据清洗 / TSDB 落库 / Redis 实时状态 / 特征窗口 / 异步推理

存储层：
  MySQL 存业务结果
  TSDB 存高频遥测
  Redis 存实时状态、在线状态、幂等控制、预警抑制
```

启动入口在：

```text
backend/app/main.py
```

路由注册入口在：

```text
backend/app/api/routes.py
```

---

### `POST /api/v1/warnings/{warning_id}/status`

功能：

- 修改预警状态。
- 支持操作人和备注。
- 使用严格状态机控制，不能随意跳转。

请求体示例：

```json
{
  "status": "acknowledged",
  "operator": "系统操作员",
  "note": "已确认"
}
```

当前允许的状态迁移：

```text
new
  → acknowledged
  → processing
  → resolved

new
  → ignored

acknowledged
  → ignored

processing
  → ignored
```

不允许的状态迁移会返回 `409`。

例如：

```text
resolved → processing
ignored → new
new → resolved
```

对应文件：

```text
backend/app/api/v1/warnings.py
backend/app/repositories/maintenance_repository.py
```

## 8. 实时监测接口

### `GET /api/v1/realtime/overview`

功能：

- 获取实时监控总览。
- 主链路面向真实工业设备：设备/网关通过 MQTT 发布到 EMQX，桥接进入 Kafka raw，清洗后写入 TSDB 和 Redis。
- 主要从 Redis 读取设备最新快照。
- 同时合并 MySQL 中最近预测和预警。

返回结构示例：

```json
{
  "devices": [],
  "online_total": 0,
  "device_total": 0,
  "warning_total": 0,
  "prediction_total": 0
}
```

每台设备会包含：

```text
设备编号
最新点位
最新数值
单位
质量分数
时间戳
在线状态
最新预测
最新预警
```

---

### `GET /api/v1/realtime/devices/{device_code}`

功能：

- 获取单台设备实时详情。

数据来源：

```text
Redis：设备最新快照、在线状态
TSDB：最近各点位最新值
MySQL：最新预测、最新预警
```

返回内容：

```text
设备编号
最新快照
最新遥测点位
最新预测
最新预警
```

对应文件：

```text
backend/app/api/v1/realtime.py
backend/app/realtime/overview.py
backend/app/realtime/device_snapshot.py
backend/app/tsdb/telemetry_repository.py
```

## 10. 数据导入与模型训练接口

### `POST /api/v1/ingestion/ai4i`

功能：

- 上传 AI4I CSV 数据集。
- 训练真实模型套件。
- 可选择是否回放训练展示数据，生成预测和预警；该回放服务于页面展示与模型结果联调，不替代实时工业设备检测链路。

表单参数：

```text
file: CSV 文件
replay_demo_data: bool
```

---

### 模式一：只训练

```text
replay_demo_data = false
```

执行链路：

```text
CSV
  → 解析数据
  → 训练模型套件
  → 保存 active 模型
  → 保存模型指标
```

不会生成预测和预警。

---

### 模式二：训练并回放

```text
replay_demo_data = true
```

执行链路：

```text
CSV
  → 解析数据
  → 训练模型套件
  → 保存 active 模型
  → 写入设备
  → 写入传感器读数
  → 生成特征窗口
  → 模型推理
  → 写入预测
  → 中高风险生成预警
```

返回示例：

```json
{
  "status": "completed",
  "mode": "train_and_replay",
  "batch_id": 1,
  "imported_rows": 1000,
  "trained_rows": 1000,
  "replay_enabled": true,
  "prediction_count": 1000,
  "warning_count": 123
}
```

对应文件：

```text
backend/app/api/v1/ingestion.py
backend/app/ingestion/ai4i.py
backend/app/models/model_suite.py
backend/app/models/registry.py
backend/app/services/maintenance.py
```

## 11. 遥测接入接口

### `POST /api/v1/telemetry/readings`

功能：

- 接收 HTTP 实时遥测数据。
- 该接口用于开发调试、网关旁路或第三方系统适配；生产主链路仍是真实设备 MQTT→EMQX→Kafka。
- 当前设计不是直接推理，而是转换成点位事件后写入 Kafka raw topic。
- 返回状态码 `202 Accepted`。

请求示例：

```json
{
  "device_code": "CNC-001",
  "device_name": "一号机床",
  "device_type": "cnc",
  "recorded_at": "2026-07-09T09:30:00Z",
  "readings": [
    {
      "sensor_code": "spindle_temperature",
      "value": 62.4,
      "unit": "C"
    }
  ]
}
```

内部转换：

```text
TelemetryPayloadIn
  → 多个 TelemetryEvent
  → Kafka: factory.telemetry.raw
```

返回示例：

```json
{
  "status": "accepted",
  "mode": "async_raw_ingestion",
  "device_code": "CNC-001",
  "accepted_events": 1,
  "raw_topic": "factory.telemetry.raw",
  "next_stages": [
    "factory.telemetry.cleaned",
    "factory.features.windowed",
    "factory.predictions.created"
  ]
}
```

如果 Kafka 写入失败，会返回 `503`。

---

### `POST /api/v1/telemetry/mqtt/simulate`

功能：

- 作为实时页展示/联调适配器使用。
- 后端临时扮演 MQTT client，将同一份标准遥测 payload 发布到 EMQX。
- 该入口只用于演示、联调和无真实网关时的链路验证，不能替代真实工业设备接入语义。

链路边界：

```text
展示/联调请求
  → 后端 MQTT client
  → EMQX
  → Kafka: factory.telemetry.raw
```

生产主链路仍为：

```text
真实设备 / 工业网关
  → MQTT topic
  → EMQX
  → Kafka
  → TSDB / Redis
  → 特征窗口 / 推理 / 预警
```

---

### `WebSocket /api/v1/telemetry/ws/readings`

功能：

- 通过 WebSocket 接收实时遥测 JSON。
- 用于调试、演示长连接上报和前后端联调；不作为真实工业设备主链路定义。
- 每条消息解析后同样写入 Kafka raw topic。

成功响应：

```json
{
  "ok": true,
  "data": {}
}
```

失败响应：

```json
{
  "ok": false,
  "detail": "错误原因"
}
```

对应文件：

```text
backend/app/api/v1/telemetry.py
backend/app/ingestion/http_adapter.py
backend/app/ingestion/http_schemas.py
backend/app/ingestion/mqtt_simulator.py
```

## 13. MQTT 到 Kafka raw

真实设备主链路：

```text
PLC / CNC / 采集器 / 工业网关
  → MQTT
  → EMQX
  → Kafka: factory.telemetry.raw
  → 清洗、TSDB、Redis、特征窗口、推理和预警
```

模块：

```text
backend/app/ingestion/mqtt_to_kafka.py
```

功能：

```text
真实设备 / 工业网关发布 MQTT topic
  → EMQX 接收标准遥测消息
  → MQTT bridge / 后台订阅器转入 Kafka raw topic
```

该链路是工业实时检测的生产主链路；HTTP、WebSocket 和 `/telemetry/mqtt/simulate` 只承担调试、展示或适配职责。

相关配置：

```text
mqtt_to_kafka_enabled
mqtt_broker_host
mqtt_broker_port
mqtt_telemetry_topic
kafka_telemetry_raw_topic
```

默认状态：

```text
mqtt_to_kafka_enabled = False
```

## 22. 当前默认配置状态

虽然后端已经具备完整的异步管道代码，但默认配置下后台消费者不会自动运行。

默认关闭项：

```text
mqtt_to_kafka_enabled = False
raw_telemetry_consumer_enabled = False
cleaned_telemetry_consumer_enabled = False
feature_consumer_enabled = False
inference_consumer_enabled = False
```

这意味着：

```text
默认启动后端：
  REST API 可用
  WebSocket 接口可用
  AI4I 导入训练可用
  预测 / 预警查询可用

但完整实时链路不会自动跑：
  MQTT → Kafka → TSDB → Redis → 特征 → 推理
```

如果要跑完整链路，需要在 `.env` 中开启这些配置，并确保 Kafka、Redis、TSDB、MQTT/EMQX 服务可用。

## 23. 当前后端功能边界总结

### 已具备能力

```text
1. FastAPI REST API
2. 健康检查
3. 设备列表
4. 首页汇总
5. AI4I CSV 导入训练
6. active 模型保存 / 查询 / 重置
7. HTTP 实时遥测接入
8. WebSocket 实时遥测接入
9. Kafka raw / cleaned / feature / inference 异步链路
10. TSDB 写入和查询
11. Redis 实时快照和在线状态
12. Redis 幂等控制
13. Redis 预警抑制
14. 预测结果查询
15. 预警查询
16. 预警状态流转
17. 维护建议生成
```

### 当前重点限制

```text
1. 后台消费者默认关闭
2. 遥测接口依赖 Kafka，可用性取决于 Kafka 是否启动
3. active 模型必须先通过 AI4I CSV 导入训练生成
4. 实时详情依赖 Redis 和 TSDB 中已有数据
5. MySQL 仍保留 sensor_readings 兼容演示，但真实高频数据主链路应走 TSDB
```

## 24. 一句话概括

当前后端已经形成了一个工业设备预测性维护数据底座：

```text
接口层负责业务访问
Kafka 负责流式解耦
TSDB 负责高频遥测
Redis 负责实时状态
MySQL 负责业务结果
模型层负责故障概率、健康评分、风险等级和维护建议
```

更准确地说，它现在的目标架构已经接近：

```text
机床 / PLC / 工业网关
  → MQTT / HTTP / WebSocket
  → Kafka raw
  → 数据清洗
  → Kafka cleaned
  → TSDB + Redis
  → 特征窗口
  → 异步推理
  → MySQL 预测与预警
  → 前端实时监测与预警处置
```
