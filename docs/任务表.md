# 工业设备预测性维护系统任务表

## 当前架构分层

| 层级 | 目录 | 职责 |
| --- | --- | --- |
| 数据接入层 | `backend/app/ingestion` | 真实设备 MQTT 遥测接入、HTTP/WS 调试入口、展示/联调适配器、设备基础信息接入 |
| 数据治理层 | `backend/app/governance` | 缺失值处理、异常值处理、平滑、标准化、窗口切分前的数据质量控制 |
| 大数据计算层 | `backend/app/compute` | 离线特征、实时聚合、设备状态统计、历史趋势分析 |
| 模型分析层 | `backend/app/models` | 故障预测、异常检测、健康评分、RUL、模型评估 |
| 业务服务层 | `backend/app/services` | 设备管理、预测分析、风险预警、维护建议、报表逻辑 |
| 可视化展示层 | `backend/app/api` + `frontend/src` | API 边界、监控大屏、设备详情、预警中心、模型管理页面 |

## 当前建议路线

先用 `AI4I 2020 Predictive Maintenance` 快速落地完整系统闭环；等业务流跑通后，再决定是否替换为 NASA C-MAPSS 做 RUL，或者引入深度模型做论文亮点。


 可以，给你一份按“真实工厂能落地”的架构清单，不讲空话，直接按上线所需来。

  一、总体架构分层

  - 设备层：PLC、CNC、DCS、传感器、振动采集器、电表、温度压力流量设备。
  - 边缘接入层：工业网关、协议采集服务、断点续传、本地缓存、离线补传。
  - 平台接入层：MQTT/OPC UA/Modbus TCP/HTTP 统一接入，消息标准化。
  - 数据处理层：数据清洗、时间对齐、异常值处理、窗口聚合、特征生成。
  - 模型服务层：在线推理、批量推理、模型版本管理、灰度切换。
  - 业务应用层：实时监测、设备健康、故障预测、预警中心、工单联动、报表。
  - 运维治理层：权限、审计、日志、监控、告警、备份、容灾。

  二、必须具备的系统模块

  - 设备主数据：设备台账、设备编码、产线/车间/工厂层级、关键部件映射。
  - 点位主数据：传感器编码、单位、采样频率、量程、上下限、所属设备。
  - 实时接入服务：持续接收设备数据，不能靠人工点按钮发送。
  - 历史数据导入：支持 CSV、数据库、MES/SCADA 导入，用于冷启动和回补。
  - 数据质量治理：缺失补值、重复去重、异常值裁剪、时间戳纠偏、单位统一。
  - 特征窗口服务：按设备持续生成 1min/5min/15min 等窗口特征。
  - 在线预测服务：输出故障概率、健康评分、RUL、异常原因、解释信息。
  - 规则引擎：把模型结果转成风险等级、预警条件、联动动作。
  - 预警中心：预警生成、确认、升级、关闭、处理记录。
  - 设备状态服务：在线、离线、停机、检修、故障、风险上升。
  - 工单/维修联动：对接 CMMS/EAM/MES，形成真实闭环。
  - 模型管理：训练、发布、回滚、版本对比、指标留档。
  - 可观测性：接口监控、消息堆积、推理耗时、设备掉线率、预警命中率。

  三、真实工厂建议的数据架构

  - 热数据：MySQL/PostgreSQL，存设备主数据、最新状态、预警、工单、模型版本。
  - 时序数据：InfluxDB、TimescaleDB 或 ClickHouse，存高频传感器数据。
  - 消息总线：MQTT 负责设备侧轻量接入，Kafka 负责平台内部解耦和回放。
  - 冷数据：对象存储或 HDFS/MinIO，存原始文件、训练样本、模型产物、审计留档。
  - 缓存：Redis，存实时状态快照、告警抑制、幂等键、短期特征缓存。

  四、真实工厂建议的服务清单

  - gateway-service：协议适配、边缘采集、断网缓存。
  - ingestion-service：统一接入、验签、标准化、入总线。
  - quality-service：数据治理和质量评分。
  - feature-service：滑窗聚合和特征生成。
  - inference-service：在线模型推理。
  - rule-service：风险分级、预警生成、联动策略。
  - device-service：设备台账、状态、拓扑关系。
  - warning-service：预警生命周期管理。
  - workorder-adapter：对接 EAM/CMMS。
  - model-registry-service：模型注册、发布、回滚。
  - report-service：日报、周报、月报、设备健康排行。
  - ops-service：权限、审计、系统配置、监控。

  五、部署架构建议

  - 车间侧部署边缘网关，先采集再上传，不要让设备直连中心平台。
  - 厂内中心部署接入、消息、时序库、推理服务，保证低时延。
  - 集团级可再加一层中心平台，做多工厂汇总、训练、报表和模型下发。
  - OT 和 IT 网络隔离，中间走工业 DMZ。
  - 关键服务至少双实例，消息和数据库要做高可用。
  - 边缘侧必须支持离线运行和恢复后补传。

  六、非功能要求

  - 可用性：核心链路 7x24，接入和预警不能单点。

  - 幂等性：重复上报不能重复生成多条预测和预警。
  - 安全性：设备鉴权、接口鉴权、操作审计、分级授权。
  - 可维护性：配置中心、日志中心、健康检查、自动重启。
  - 可扩展性：新增产线、设备、传感器、模型时不需要重构主系统。

P0：必须先补的“可信闭环”
目标：让系统不像演示页面，而像一个能解释清楚的工业预测系统。

P0-1 数据闭环
当前已经有：

CSV 导入 → 训练模型 → 保存 active 模型 → 实时推理 → 预测记录 → 预警记录 → 前端展示
但还要补强：

原始读数
  → 数据治理
  → 特征窗口
  → 模型版本
  → 推理结果
  → 风险规则
  → 预警事件
每条预警必须能追溯：

warning_event
  ├─ prediction_id
  ├─ device_code
  ├─ feature_window_id
  ├─ model_version
  ├─ failure_probability
  ├─ health_score
  ├─ risk_level
  └─ explanation
否则答辩时别人问“为什么严重风险”，你只能说“模型算的”，这不够。

P0-2 设备和点位主数据
你现在设备更多是随着数据导入/实时上报被动生成，缺少工业系统最基本的主数据。

先做两个核心表/页面：

设备台账
  ├─ device_code
  ├─ device_name
  ├─ device_type
  ├─ factory
  ├─ workshop
  ├─ production_line
  ├─ status
  └─ created_at
点位台账
  ├─ sensor_code
  ├─ sensor_name
  ├─ device_code
  ├─ unit
  ├─ sampling_frequency
  ├─ min_value
  ├─ max_value
  └─ enabled
前端至少要有：

设备管理
  ├─ 设备列表
  ├─ 最新健康评分
  ├─ 最新风险等级
  ├─ 在线/离线/检修/故障
  └─ 传感器点位列表
这是 P0，因为没有主数据，后面的边缘接入、规则、预警都没有稳定对象。

P0-3 预警生命周期
现在预警生成了，但还不像真实系统。

至少要有状态机：

new 新预警
  → acknowledged 已确认
  → processing 处理中
  → resolved 已关闭
  → ignored 已忽略
每次状态变化要留记录：

warning_event
  ├─ status
  ├─ acknowledged_at
  ├─ resolved_at
  └─ latest_action
warning_action_log
  ├─ warning_id
  ├─ from_status
  ├─ to_status
  ├─ operator
  ├─ note
  └─ created_at
这比“加一个工单系统”更优先。预警没有生命周期，系统就只是看板。

P0-4 风险规则显式化
现在风险等级来自模型输出和阈值。建议把规则从代码里抽出来，至少形成配置结构：

风险规则
  ├─ critical: failure_probability >= 0.8 OR health_score < 30
  ├─ high:     failure_probability >= 0.6 OR health_score < 50
  ├─ medium:   failure_probability >= 0.3 OR health_score < 75
  └─ low:      otherwise
前端展示时也要能解释：

严重风险
原因：
- 故障概率 0.87，超过严重阈值 0.80
- 健康评分 26.4，低于严重阈值 30
- 主要贡献特征：Tool wear、Torque
这是让系统可信的关键。

P1：从“单机演示”升级到“准生产链路”
目标：让实时接入不再靠按钮模拟，而是形成平台接入链路。

P1-1 Kafka 真正启用
你现在有 Kafka/MQTT 接口位，但更像“预留”。

P1 要做成真实链路：

HTTP/MQTT/WebSocket
  → ingestion-service
  → Kafka topic: factory.telemetry.raw
  → quality consumer
  → Kafka topic: factory.telemetry.cleaned
  → feature consumer
  → Kafka topic: factory.features.windowed
  → inference consumer
  → prediction_logs / warning_events
即使代码还在一个 FastAPI 项目里，也要先把模块边界做成这种结构。

推荐先用模块化单体：

backend/app/
  ingestion/
    http_adapter.py
    mqtt_adapter.py
    kafka_producer.py
  quality/
    cleaner.py
    validator.py
  features/
    window_builder.py
    feature_store.py
  inference/
    predictor.py
    explainer.py
  rules/
    risk_policy.py
    warning_policy.py
  warnings/
    lifecycle.py
    repository.py
先别急着真的拆服务。先让职责边界稳定。

P1-2 Redis 真正使用
Redis 不要为了用而用。它适合放这几类东西：

Redis
  ├─ 设备在线状态
  ├─ 最新设备快照
  ├─ 告警抑制键
  ├─ 幂等键
  └─ 短期特征缓存
典型结构：

device:SIM-0001:latest
  ├─ last_seen_at
  ├─ health_score
  ├─ risk_level
  └─ status
idempotency:telemetry:{device_code}:{timestamp_hash}
warning:suppress:{device_code}:{risk_type}
这样可以解决：

重复上报重复预警
在线/离线状态不准确
每次刷新都查数据库
高频数据写 MySQL 压力过大
P1-3 时序库接入
当前 MySQL 可以撑住演示，但真实传感器数据不应该长期塞 MySQL。

推荐演进：

MySQL
  ├─ 设备
  ├─ 点位
  ├─ 预测记录
  ├─ 预警记录
  ├─ 模型版本
  └─ 用户权限
TimescaleDB / InfluxDB / ClickHouse
  └─ 高频传感器读数
MinIO / 本地对象存储
  ├─ CSV 原始文件
  ├─ 模型文件
  ├─ SHAP 解释产物
  └─ 训练数据快照
如果是毕设，我更建议 TimescaleDB，因为它和 PostgreSQL 体系更接近，解释成本低。

P2：业务系统化
目标：让系统从“预测系统”变成“运维工作台”。

P2-1 工单联动
不一定真的接 CMMS/EAM，可以先做内部轻量工单。

warning_event
  → create work_order
  → assign maintainer
  → process
  → resolve
  → feedback
工单结构：

work_order
  ├─ warning_id
  ├─ device_code
  ├─ priority
  ├─ assignee
  ├─ status
  ├─ maintenance_action
  ├─ result
  └─ closed_at
这一步的价值很高，因为它让系统形成：

预测 → 预警 → 维修 → 反馈 → 模型评估
P2-2 用户、角色、权限、审计
真实系统必须有：

用户
  ├─ 系统管理员
  ├─ 产线管理员
  ├─ 运维人员
  └─ 只读评审/领导
权限
  ├─ 查看设备
  ├─ 确认预警
  ├─ 关闭预警
  ├─ 导入数据
  ├─ 发布模型
  └─ 修改规则
审计日志：

audit_log
  ├─ operator
  ├─ action
  ├─ resource_type
  ├─ resource_id
  ├─ before
  ├─ after
  └─ created_at
这块不要 P0 做，否则会拖慢主线。P2 合适。

P2-3 模型发布、回滚、灰度
当前 active model 是保存和加载，够演示，但不够生产。

目标状态：

model_registry
  ├─ candidate 候选
  ├─ active 当前
  ├─ archived 归档
  └─ rollback 回滚目标
发布流程：

训练完成
  → 生成 candidate
  → 指标对比
  → 人工确认
  → 发布 active
  → 保留旧版本
灰度可以简单做：

10% 设备使用新模型
90% 设备使用旧模型
毕设里不一定真做灰度，但要把结构设计出来。

P3：生产级/集团级能力
目标：从“单厂可用”升级到“多厂、多产线、可运营”。

P3-1 多工厂多租户
数据结构要从一开始预留：

factory
  └─ workshop
      └─ production_line
          └─ device
              └─ sensor
所有核心表最好逐步加：

factory_id
workshop_id
line_id
但不建议 P0 就大改。可以 P1/P2 逐步补。

P3-2 高可用和监控
包括：

Prometheus / Grafana
  ├─ 接口耗时
  ├─ 推理耗时
  ├─ Kafka 堆积
  ├─ 设备掉线率
  ├─ 预警生成数
  └─ 模型错误率
以及：

健康检查
  ├─ /health
  ├─ /ready
  └─ /metrics
P3-3 数据湖和集团级训练
这个是长期，不适合现在做。

多工厂数据
  → 数据湖
  → 统一训练
  → 模型评估
  → 模型下发
可以写进论文展望，不一定实现。

我对你当前项目的判断
你现在最不应该继续堆 UI 页面。

当前最该做的是：

数据链路可信
  > 预警生命周期
